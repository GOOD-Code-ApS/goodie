# @goodie-ts/hono

HTTP controller decorators and Hono integration for goodie-ts. Provides route metadata decorators, a transformer plugin for compile-time route wiring, and runtime server infrastructure.

## Key Files

| File | Role |
|------|------|
| `src/metadata.ts` | `HONO_META` symbol keys, `ControllerMetadata`, `RouteMetadata` types |
| `src/controller.ts` | `@Controller(basePath?)` — marks class as HTTP controller, stores basePath |
| `src/route.ts` | `@Get`, `@Post`, `@Put`, `@Delete`, `@Patch` — method decorators via `createRouteDecorator()` |
| `src/validate.ts` | `@Validate({ json: schema })` — attaches validation metadata for zod-validator middleware |
| `src/server-config.ts` | `ServerConfig` — `@ConfigurationProperties('server')` bean with `host` and `port` defaults |
| `src/embedded-server.ts` | `EmbeddedServer` — `@Singleton` that wraps `@hono/node-server`, uses `ServerConfig` for defaults |
| `src/plugin.ts` | Transformer plugin — generates `createRouter()` and `startServer()` from controller metadata |
| `src/index.ts` | Public exports |

## HONO_META Keys (metadata.ts)

All are Symbols under the `HONO_META` object:
- `CONTROLLER` — `ControllerMetadata` (`{ basePath }`)
- `ROUTES` — `RouteMetadata[]` (`{ method, path, methodName }`)

## Transformer Plugin (`src/plugin.ts`)

The hono plugin is auto-discovered at build time via `"goodie": { "plugin": "dist/plugin.js" }` in package.json. It reads `bean.metadata.controller` (set by the resolver during scan) and generates:

- **`createRouter(ctx: ApplicationContext): Hono`** — retrieves controllers from DI context, registers routes on a Hono instance
- **`startServer(options?)`** — starts the ApplicationContext, creates the router, and calls `EmbeddedServer.listen()`

The plugin does NOT live in the transformer — this keeps `@goodie-ts/transformer` framework-agnostic. Route codegen is fully owned by this package.

## ServerConfig + EmbeddedServer

`ServerConfig` is a library bean shipped via `beans.json`. It uses `@ConfigurationProperties('server')` to bind from JSON config files or env vars:

```json
{ "server": { "host": "localhost", "port": 3000 } }
```

`EmbeddedServer` is a `@Singleton` that takes `ServerConfig` as a constructor dependency. The generated `startServer()` calls `EmbeddedServer.listen(router, options)` — options override config defaults.

## Design Decisions

- **`@Controller` implies singleton** — cannot combine with `@Singleton`, `@Injectable`, or `@Module` (throws `InvalidDecoratorUsageError`)
- **Methods receive Hono `Context` directly** — no parameter decorator magic (Stage 3 has no param decorators)
- **Return value conventions**: `Response` passthrough, `undefined`/`null` returns 204, everything else is `c.json(result)`
- **Plugin-based codegen** — route wiring is generated by this package's plugin, not by the transformer core. This follows the Micronaut pattern where framework modules contribute their own code generation.
- **Controller metadata on beans** — the resolver stores controller route data on `bean.metadata.controller`, allowing any plugin to read it without transformer API changes

## Testing

- **Transformer tests** (`packages/transformer/__tests__/controller-pipeline.test.ts`) — verify controller metadata is stored on `bean.metadata` by the resolver
- **Hono tests** (`packages/hono/__tests__/hono-plugin.test.ts`) — verify the plugin generates correct `createRouter`/`startServer` code
- No cross-package test dependencies

## Gotchas

- Route decorators are matched by name only (no import source verification), but only scanned on `@Controller` classes
- Controller variable names in generated code use collision-safe naming (`className:importPath` keying)
- `@Validate` generates `zValidator()` middleware — requires `@hono/zod-validator` and `zod` as peer deps
